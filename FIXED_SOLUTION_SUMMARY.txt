ROOT CAUSES IDENTIFIED
================================================================================

ROOT CAUSE #1: Type Conversion Failure
  
  Problem: CLIPS (read) function returns atoms, not FLOAT/INTEGER
  Example: (read) on input "145.0" returns atom/symbol, not float 145.0
  Impact: Template type validation fails
    (patient (fpg "145.0")) ← WRONG (atom, not float)
    (patient (fpg 145.0)) ← CORRECT (float)
  
  Evidence: Template expects (fpg (type FLOAT)) but receives atom
  
ROOT CAUSE #2: Duplicate Output Rules

  Problem: Rules fire multiple times for same pattern match
  Example: If diagnosis exists, output rule fires every cycle
  Impact: Output repeats infinitely or many times
  
  Evidence: No guard condition to prevent re-firing
  
ROOT CAUSE #3: Incomplete Error Handling

  Problem: Error output rule not protected
  Example: Error message repeats when error phase active
  Impact: Error spam in output
  
  Evidence: output-validation-error had no guard condition

================================================================================
SOLUTIONS IMPLEMENTED
================================================================================

FIX #1: Type Conversion with integer() and float()

  BEFORE (BROKEN):
    (bind ?fpg (read))
    (assert (patient (fpg ?fpg)))
    ↓
    Problem: ?fpg is atom, template expects FLOAT

  AFTER (FIXED):
    (bind ?fpg-raw (read))
    (bind ?fpg (float ?fpg-raw))
    (assert (patient (fpg ?fpg)))
    ↓
    Solution: ?fpg properly converted to FLOAT type

  Applied to: Patient ID (integer), FPG (float), HbA1c (float)

FIX #2: Guard Facts to Prevent Duplicate Output

  BEFORE (BROKEN):
    (defrule output-rule
      (fact1)
      (fact2)
      =>
      (printout t "Output" crlf))
    ↓
    Problem: Fires every cycle while facts exist

  AFTER (FIXED):
    (defrule output-rule
      (fact1)
      (fact2)
      (not (shown-rule))  ← GUARD: only if not yet shown
      =>
      (assert (shown-rule))  ← SET FLAG: prevent future fires
      (printout t "Output" crlf))
    ↓
    Solution: Guard prevents re-firing

  Applied to:
    • output-diagnosis-display (guard: shown-diagnosis)
    • output-validation-error (guard: shown-error)

FIX #3: Error Handling with Guards

  Same guard pattern applied to error output rule:
    (defrule output-validation-error
      (system-state (phase error))
      (not (shown-error))
      =>
      (assert (shown-error))
      ...)

TECHNICAL DETAILS - WHY IT WORKS NOW
================================================================================

Type System:

  CLIPS templates require specific types:
    (deftemplate patient
      (slot id (type INTEGER))      ← Must be INTEGER
      (slot fpg (type FLOAT))        ← Must be FLOAT
      (slot hba1c (type FLOAT)))     ← Must be FLOAT

  (read) returns atoms:
    (read) on "145.0" → atom 145.0 (not float)
    (read) on "1001"  → atom 1001 (not integer)

  Fix: Explicit conversion
    (float "145.0") → 145.0 (type FLOAT)
    (integer "1001") → 1001 (type INTEGER)

Guard Pattern:

  Problem: Rules match indefinitely while pattern exists
    Fact in memory → Rule fires
    Fact still in memory → Rule fires again
    Fact STILL in memory → Rule fires AGAIN

  Solution: Add flag to prevent re-matching
    First match: Condition true, asserts flag
    Second match: Condition false (flag exists), rule doesn't fire
    Third match: Condition still false, rule doesn't fire
